# Operational Evidence and Lineage

The operational evidence layer converts synthetic commerce and analytical artifacts into a deterministic, source-bound incident evidence package. It is designed for later use by governed tools and an investigating agent, but contains no language-model calls and exposes no database credentials.

## Evidence surfaces

The artifact contains:

- structured service-health windows for checkout, payments, pipelines, fulfilment, and seller feeds,
- deployment, configuration, and feature-flag history,
- a canonical evidence catalog with payload hashes, trust levels, and evidence roles,
- a directed acyclic lineage graph,
- curated runbooks and historical incidents,
- a chronological incident timeline,
- source-artifact lineage to dataset, analytics, detection, and impact manifests,
- deterministic integrity and semantic-reconstruction checks.

Evidence roles are explicit: `supporting`, `contradictory`, or `context`. They indicate investigative relevance, not proven causality. These labels belong to the synthetic benchmark configuration and are not generated by an LLM.

## Build

```bash
paic evidence build \
  --dataset-dir data/generated/impact-source-smoke \
  --impact-dir data/generated/impact-smoke \
  --config configs/evidence/smoke.yaml \
  --output-dir data/generated/evidence-smoke

paic evidence validate \
  --evidence-dir data/generated/evidence-smoke \
  --dataset-dir data/generated/impact-source-smoke \
  --impact-dir data/generated/impact-smoke
```

## Safety boundary

This capability assembles evidence only. Query policy, SQL parsing, tool permissions, approval gates, and LLM orchestration are intentionally separate capabilities. All retrieved text must later be treated as untrusted input.
